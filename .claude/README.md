# VaultPay — AI Test Infrastructure

## Skills (user-invocable)

Type skill commands directly into the Claude Code prompt (CLI or IDE extension). Claude Code recognises the `/` prefix and executes the matching skill.

| Skill | Command | What it does |
|---|---|---|
| test-plan | `/test-plan [feature]` | Reads references, drafts TC scenarios for the feature, saves to `docs/test-plans/{feature}.md` |
| generate-tests | `/generate-tests [feature or TC IDs]` | Reads page objects + flows + test plan (optional), writes `tests/e2e/tests/{feature}.spec.ts`, runs Playwright, self-reviews |

**Recommended workflow** — run `/test-plan` first, then `/generate-tests`:
```
/test-plan send-money        →  docs/test-plans/send-money.md
/generate-tests send-money   →  tests/e2e/tests/send-money.spec.ts (run + reviewed)
```

**`/generate-tests` without a test plan** — also supported. The skill proceeds from the user's description and saves a test plan to `docs/test-plans/` at the end. Use this for ad-hoc or iterative test development.

## References

Reference files contain domain knowledge — facts and rules that are not derivable from reading the code.

| File | What it contains | Read by |
|---|---|---|
| `domain.md` | App URL, credentials, tech stack, all mock data (users, transactions, cards, budget, contacts), timing delays | `/test-plan`, `/generate-tests` |
| `business-rules.md` | Every exact validation message, toast string, and visual threshold used in assertions | `/test-plan`, `/generate-tests` |
| `test-planning.md` | TC format spec, TC ID prefix table (Auth-TC-001 etc.), category/priority definitions, coverage checklist | `/test-plan` |
| `test-spec-template.md` | POM-based Playwright code structure, async wait patterns, assertion examples, anti-patterns | `/generate-tests` |
| `review-checklist.md` | Quality checklist run as the final step of `/generate-tests` before declaring done | `/generate-tests` |
| `user-flows.md` | Simulated delays, nav mapping, validation error tables, known limitations | `/test-plan` |
| `ui-selectors.md` | All `data-testid` values by page — **planning context only** | `/test-plan` |

> **Selector authority**: `tests/e2e/pages/*.ts` page objects are the authoritative source for selectors at implementation time. `/generate-tests` reads page objects directly — not `ui-selectors.md`. If a testid changes, update the page object; the skill picks it up automatically.
